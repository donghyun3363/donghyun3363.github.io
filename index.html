새로운 공간 ' Fam ' 을 소개합니다

'가족' 세상에서 가장 소중한 공간입니다.

바쁜 일상때문에 그 공간 잃어버리고 있지 않나요?

그 공간을 ' Fam '이 채워드리겠습니다.

Download App

3가지가 필요했습니다.


Fam은 가족에게 3가지 공간을 제공합니다.

소통 공간

오늘의 하루가 어땠는지 혹은 무얼 했는지를 사진과 함께 글과 댓글로 대화를 나누는 공간이 필요했습니다.

추억 공간

추억은 사진으로 남는 것을 알 것입니다. 아날로그 앨범처럼 지난 가족간의 추억 되새길 공간이 필요했습니다.

버킷 공간

가족 구성원들이 당신에게 무얼 원하는지 아나요? 그래서 버킷을 함께 이뤄나갈 공간이 필요했습니다.

소통 공간

메신저처럼 실시간으로 곧장 소통을 나눠봐요!



글과 글쓰기 기능

'라이크' 눌러 하트를 서로 건네주세요!

댓글과 댓글쓰기 기능

대문 타이틀 및 사진 변경

그룹원 초대 기능

회원, 초대 등 그 외의 기능들을 보여줍니다.

Technology

앱 개발 시 사용된 기술들의 간단한 소개입니다.

FireBase DB 및 Storage

  • 실시간 Messaging 기법을 위해 polling 방식인 child Listener를 지원하는 FireBase Nosql DB을 사용하였습니다.

  • polling signal에 따라 DB의 추가, 변경, 삭제 등의 값을 인지하여 여러 기능을 용이하게 사용하였습니다.

  • Transition Handler를 사용하여 DB 데이터 접근 시 데이터 손실을 방지하도록 하였습니다.

  • Firebase storage의 Glide를 이용하여 이미지 로딩 및 캐싱을 통해 사진의 업로드의 효율을 높였습니다.

    • FirBase Auth 및 FireBase FCM

  • Authorization을 사용하여 메일 인증, 자동 로그인 등을 구현하였습니다.

  • FCM을 구현하여 회원들에게 필요한 정보 메시지를 보내도록 구현하였습니다.

    • FireBase Dynamic Link 및 Invite

  • Dynamic Link를 통해 웹 사용자가 해당 앱을 손 쉽게 다운 받을 수 있도록 하였습니다.

  • Invite의 기능으로 메일 및 SMS로 그룹원을 초대하도록 하였습니다.

    • RecyclerView, Adapter

  • SNS 앱이라는 점에 9개의 리사이클러뷰가 사용이 되었습니다. 동적인 데이터를 효율적으로 수행되기 위해 어댑터를 커스텀하여 사용하였습니다.

  • 동적으로 발생하는 데이터로 인해 리플래쉬 되는 어댑터 과정을 최소화하기 위해 notify 적정 사용으로 재사용을 줄이도록 구현하였습니다.

    • Async, DataBus

  • 저장소로부터 네트워킹 작업에 대한 UI Thread 작업 손실을 막기위해 Async를 사용하였습니다.

  • DataBus를 통해 화면간의 유기적인 데이터의 이동으로 데이터 공유를 하도록 하였습니다.

    • 버그를 잡는 디버깅의 끈기

  • SNS 특성 상 동적인 데이터가 많이 발생하기 때문에 그 만큼 실수로 인한 버그가 많았습니다. 이를 끈기로 해결하고자 했습니다.

  • 버그의 최소화를 위해 Json형태로 저장되는 Nosql의 DB의 구조화를 간편화하게하여 데이터 중복 및 손실을 최소하려 힘썼습니다.

    • Trouble Shooting

    s

    " FireBase SingleValueEvent를 사용하여 DataBase로부터 동기화 데이터를 가져왔습니다. "

    "사용자에 의한 지속적인 데이터 변화를 감당하기에 Once Polling 방식으로는 처리할 로직들이 늘어나고 매우 복잡해졌습니다."

  • 해결

    • => 파이어베이스에서 제공하는 Continuous polling 방식 Handler를 사용하여 데이터 변화에 대한 감지하여 로직을 간편화

  • 주의

    • => DB의 참조위치를 잘 못 고려할 경우, 원치 않는 데이터 변화에 따른 client가 Interrupt를 받는 오류가 생길 수 있습니다.

    많은 데이터의 변화

    Firebase Database

    "UI에 대한 고민, 아이폰 핸드폰 사용자로 안드로이드 UI에 대한 이해부족으로 인한 App의 싱거움이 느껴졌습니다."

  • 해결

    • => 머티리얼 디자인의 Document를 참고한 후 전체적인 UI를 싹 바꿨습니다. 위젯의 계층 구조를 지키며, 엘리베이션 높이 그리고 색상의 어울림을 고려했습니다.

    UI 유저인터페이스

    Google

    "Storage로부터 연속적으로 가져올 경우 UI 스레드의 높아지는 복잡성과 병행처리 문제"

  • 해결

    • => 사용자가 Storage로 부터 데이터를 기다리는 동안 다른 UI를 처리할 경우 에러가 발생하곤 했습니다. Async 병렬처리를 통해 위와 같은 문제를 해결하고 좀 더 빠른 데이터 처리와 간편한 UX를 처리할 수 있었습니다.

    병렬 처리

    AsyncTask

    "사용자로부터 데이터가 변할 경우, 리사이클러 뷰의 아이템을 추가 및 삭제, 변경이 되고 notify 시 뷰를 다시 그리는 작업을 합니다. 하지만, 전부 그리는 경우로 인해 앱의 무거움이 생겼습니다."

  • 해결

    • => notifyDataSetChanged로 리사이클러뷰를 지속적으로 갱신하여 원하는 데이터만이 아닌 뷰를 모두 다시 그리는 작업을 했던 것이 문제였습니다. 서버로부터 들어오는 item의 key를 ArrayList에 저장하고, 원하는 데이터에 대해서만 index를 주고 notify하여 해결했습니다. 코드에 대해서 좀 더 복잡해지기는 하지만, 리사이클러뷰의 생산성에 효율적인 처리였습니다.
